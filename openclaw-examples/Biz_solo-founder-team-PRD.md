# Solo Founder Virtual Team 产品需求文档（PRD）

## 1. 文档信息
- **产品名称**：Solo Founder Virtual Team
- **产品定位**：基于 OpenClaw 构建的单人创业者虚拟多智能体协作系统
- **文档版本**：v1.0
- **文档状态**：初版
- **编写角色**：产品经理
- **发布日期**：2026-04-08

## 2. 项目背景
Solo Founder 在创业早期通常同时承担产品、研发、市场、销售、运营、客户支持等多重职责，面临以下核心问题：
- 决策与执行均集中于个人，时间与注意力严重稀缺
- 跨职能工作切换频繁，导致效率下降与上下文丢失
- 单一 AI 助手虽可回答问题，但难以形成稳定的“团队分工”与持续协作机制
- 多任务并行时，缺乏可靠的任务拆解、执行隔离与结果回收能力

OpenClaw 提供了多会话、多智能体、共享工作区、自动化调度与长期记忆等基础设施，具备构建“虚拟团队”的技术条件。基于此，Solo Founder Virtual Team 的目标是让创始人通过单一入口与“主智能体”交互，由系统自动拆解任务、派发给不同职能子智能体执行，并通过共享文件系统进行协作与交付，从而显著提升个人创业效率。

## 3. 产品愿景
为 Solo Founder 提供一个像“虚拟核心团队”一样工作的多智能体系统，使创始人以单一对话入口即可驱动多职能并行执行，实现从想法、规划、执行到反馈的闭环。

## 4. 产品目标

### 4.1 业务目标
1. 降低 Solo Founder 的跨职能切换成本
2. 提升复杂任务的并行处理能力与交付速度
3. 让 AI 从“问答工具”升级为“可调度的虚拟团队”
4. 建立可持续的日常自动化经营机制，如提醒、巡检、汇报、定时任务执行

### 4.2 用户目标
1. 用户只需通过 Telegram 与一个主入口沟通，无需分别管理多个 Agent
2. 用户可以像管理团队一样下达目标，而非逐步微操每一项执行细节
3. 用户可以获得清晰的进度、产物与总结，避免黑盒执行
4. 用户可以在不中断主会话的前提下，让多个专业角色并行工作

### 4.3 产品成功指标（建议）
- 任务首次响应时间 ≤ 10 秒
- 可拆解任务自动路由成功率 ≥ 85%
- 多子任务并行场景下的任务完成率 ≥ 90%
- 周期性自动任务按时触发成功率 ≥ 99%
- 用户对任务结果可用性满意度 ≥ 4.5/5
- 用户每周通过系统节省的操作时间 ≥ 10 小时

## 5. 目标用户

### 5.1 核心用户画像
- 单人创业者 / 独立开发者 / 小型工作室负责人
- 需要同时管理产品、研发、市场、销售、运营等职能
- 熟悉即时通讯工具，偏好轻量化工作入口
- 对自动化、AI 协作与效率工具有较高接受度

### 5.2 典型使用场景
- 让系统同时完成需求整理、技术方案草拟与官网文案撰写
- 安排定时跟进潜在客户、生成周报、检查指标异常
- 把一个模糊目标拆成多个职能任务并持续推进
- 在出差、碎片化时间中通过手机统一调度虚拟团队

## 6. 产品范围

### 6.1 In Scope
- Telegram 单入口对话
- 主智能体统一接收需求、理解意图、拆解任务、调度执行
- 多职能子智能体按职责执行任务
- 基于共享文件系统的任务交接与产物沉淀
- 长短期记忆管理（如 MEMORY.md）
- Cron 定时任务与 Heartbeat 主动汇报
- 结果汇总、状态同步与用户反馈闭环

### 6.2 Out of Scope（首期）
- 多人协同与复杂权限体系
- 图形化 Web 控制台
- 跨多个外部 IM 平台的统一入口
- 自动财务支付、自动对外发布等高风险无确认动作
- 企业级审计、租户隔离、多组织管理能力

## 7. 产品定位与核心价值
本产品不是一个单体对话机器人，而是一个“单入口、多角色、可调度、可记忆、可自动化”的 AI 虚拟团队系统。

其核心价值包括：
1. **统一入口**：降低用户操作复杂度
2. **角色分工**：按职能拆解问题，减少单 Agent 负载与混乱
3. **上下文隔离**：通过子会话和独立沙盒降低任务相互污染
4. **文件化协作**：以共享工作区实现稳定交接与可追溯结果沉淀
5. **自动推进**：通过 Cron 与 Heartbeat 实现“非同步待命团队”能力

## 8. 总体架构设计

### 8.1 架构概览
系统采用五层架构：

1. **交互层**
   - 人类用户通过 Telegram 发起对话
   - Gateway 负责连接 Telegram 与 OpenClaw 系统
   - 系统保持单一入口，所有请求统一进入主智能体

2. **调度层**
   - 主智能体负责理解用户意图
   - 根据任务类型做拆解、优先级判断、依赖分析
   - 通过 `sessions_spawn` 派生子智能体，建立并行或串行工作流

3. **执行层**
   - 子智能体运行在隔离执行沙盒中
   - 不同职能子智能体采用最适合的大模型与技能集
   - 典型角色包括：开发 Agent、营销 Agent、商务 Agent、研究 Agent、运营 Agent

4. **存储与通信层（FS-based IPC）**
   - 各 Agent 共享 Shared Workspace
   - 通过读写文件进行任务交接、状态同步、结果沉淀
   - 共享 `MEMORY.md` 用于长期记忆与关键决策记录
   - 用文件而非直接共享全部上下文，降低上下文污染与 token 成本

5. **自动化引擎**
   - Cron 负责定时触发任务，如日报、周报、巡检、提醒、数据整理
   - Heartbeat 负责在关键时刻主动向人类汇报进度、风险或提醒事项

### 8.2 核心设计原则
- **单入口，统一编排**
- **多角色，职责清晰**
- **上下文最小化暴露**
- **文件优先的异步协作**
- **自动化优先，但关键节点保留人工确认**

## 9. 角色定义

### 9.1 人类用户
- 提供目标、反馈、优先级与最终决策
- 查看汇报，审批高风险动作

### 9.2 主智能体（Manager Agent）
- 唯一用户接口
- 负责需求理解、任务路由、资源编排、进度汇总、对外回复
- 决定是否派生子智能体以及任务分配策略

### 9.3 子智能体（Specialist Agents）
按职责分工，可扩展但不限于：
- **开发 Agent**：需求实现、代码生成、调试、测试、重构
- **营销 Agent**：内容策划、宣传文案、社媒日历、品牌表达
- **商务 Agent**：商机跟进、合作方案、报价草案、客户沟通建议
- **研究 Agent**：竞品调研、市场信息收集、资料汇总
- **运营 Agent**：流程整理、指标追踪、周报生成、提醒与执行跟进

### 9.4 自动化代理任务
- 由 Cron 触发，按预设计划运行
- 由 Heartbeat 将关键信息主动反馈给用户

## 10. 核心业务流程

### 10.1 流程一：用户发起综合任务
**示例**：用户说“帮我准备这个新 SaaS 产品的本周推进计划，并产出 landing page 文案和开发任务拆解”。

流程如下：
1. 用户通过 Telegram 向主智能体发送需求
2. Gateway 将消息转发至 OpenClaw 主智能体
3. 主智能体解析意图，识别为复合型任务
4. 主智能体将任务拆解为：
   - 产品推进计划
   - Landing Page 文案
   - 开发任务拆解
5. 主智能体通过 `sessions_spawn` 派发给不同子智能体
6. 各子智能体在隔离沙盒内执行，并将中间产物写入 Shared Workspace
7. 主智能体读取结果、整合输出、检查一致性
8. 主智能体向用户汇总交付，并在必要时附上下一步建议或待确认事项

### 10.2 流程二：文件驱动的协作交接
1. 主智能体为任务创建工作目录或任务文件
2. 子智能体读取任务说明、约束条件与已有上下文文件
3. 子智能体生成结果文件、状态文件或建议文件
4. 其他子智能体或主智能体继续读取这些文件完成后续步骤
5. 最终产物被归档，关键结论写入 `MEMORY.md`

### 10.3 流程三：定时自动化执行
**示例**：每天 9:00 生成前一日业务摘要，每周一生成本周经营重点。

流程如下：
1. 用户或主智能体配置 Cron 规则
2. Cron 到时触发指定任务
3. 主智能体接收自动化任务并决定是否需要派生子智能体
4. 任务执行后生成摘要、提醒或报告
5. Heartbeat 在合适时间主动通知用户
6. 如任务发现异常或高风险事项，升级为待人工确认事件

### 10.4 流程四：主动汇报与异常提醒
1. 系统根据 Heartbeat 策略定期检查待办、任务进度与关键事件
2. 若存在重要更新、延迟风险、关键提醒或异常指标，主智能体主动联系用户
3. 若无显著事项，系统保持安静，避免信息骚扰

## 11. 核心功能需求

### 11.1 单入口通信
**功能描述**：用户仅需通过 Telegram 与主智能体对话。

**需求点**：
- 支持 Telegram 消息接入与稳定转发
- 用户无需了解内部多 Agent 结构
- 主智能体对外维持统一人格与上下文连续性

**验收标准**：
- 用户从 Telegram 发出的消息可稳定到达主智能体
- 系统对复杂任务可在单线程对话中完成后续编排

### 11.2 意图识别与任务路由
**功能描述**：主智能体判断用户请求属于咨询、执行、自动化、研究或跨职能协作，并进行任务拆解。

**需求点**：
- 识别任务复杂度与是否需要派生子智能体
- 支持串行、并行、依赖型任务编排
- 支持任务优先级与失败重试策略

**验收标准**：
- 复合任务可拆解为明确子任务
- 简单任务不滥用子智能体，降低调度成本

### 11.3 子智能体派生与执行隔离
**功能描述**：通过 `sessions_spawn` 创建不同职责的子智能体，在独立上下文与沙盒中工作。

**需求点**：
- 每个子智能体具备独立上下文窗口
- 支持针对职能选择不同模型与工具链
- 子智能体间默认不直接共享全部上下文

**验收标准**：
- 子智能体执行不会污染主智能体主会话
- 不同角色任务可并行运行且结果可汇总

### 11.4 Shared Workspace 文件协作
**功能描述**：通过共享文件系统实现任务交接、结果沉淀与异步协作。

**需求点**：
- 统一目录结构与命名规范
- 支持任务说明、状态、草稿、最终产物等文件类型
- 支持主智能体读取多个子智能体产物进行合并

**验收标准**：
- 文件可作为稳定中间层进行交接
- 任务全过程具备可追溯性

### 11.5 MEMORY.md 长期记忆
**功能描述**：将用户偏好、重要决策、关键事实、长期项目上下文沉淀到 `MEMORY.md`。

**需求点**：
- 支持主智能体定期更新长期记忆
- 支持从任务中提炼“值得长期保留”的信息
- 避免把临时噪音写入长期记忆

**验收标准**：
- 长期记忆有助于提升连续协作质量
- 重启后仍可恢复关键上下文

### 11.6 Cron 自动化任务
**功能描述**：按预设周期自动运行任务。

**需求点**：
- 支持日报、周报、提醒、巡检、定时研究等任务
- 支持任务模板化与参数化
- 支持失败告警与重试

**验收标准**：
- 已配置任务可按时执行
- 自动化任务产出可被用户追溯查看

### 11.7 Heartbeat 主动沟通
**功能描述**：在不打扰用户的前提下，主动汇报关键进展、提醒与风险。

**需求点**：
- 支持“何时该主动说话”的策略判断
- 避免频繁无效提醒
- 异常、临期、阻塞任务应优先通知

**验收标准**：
- 通知质量高于数量
- 用户感知为“有帮助”而非“打扰”

## 12. 关键数据与文件对象设计

### 12.1 关键对象
- **User Request**：用户原始需求
- **Task**：由主智能体拆解出的执行单元
- **Subagent Session**：对应单个子智能体执行实例
- **Artifact**：中间或最终产物文件
- **Memory Entry**：长期记忆条目
- **Automation Job**：Cron 配置的自动任务
- **Heartbeat Event**：主动汇报事件

### 12.2 建议文件结构
```text
/shared-workspace
  /tasks
    /task-001
      brief.md
      status.json
      output-dev.md
      output-marketing.md
      final-summary.md
  MEMORY.md
  /automation
    cron-jobs.md
    heartbeat-state.json
```

### 12.3 文件协作规范建议
- `brief.md`：任务目标、背景、约束、输入材料
- `status.json`：状态、负责人、更新时间、阻塞项
- `output-*.md`：不同角色的阶段性产出
- `final-summary.md`：主智能体汇总后的最终结果

## 13. 非功能需求

### 13.1 可靠性
- Gateway 与主智能体消息链路稳定可恢复
- 子任务失败不应导致全局任务崩溃
- 定时任务需要具备重试与错误记录能力

### 13.2 可扩展性
- 支持新增角色型子智能体
- 支持不同模型按任务类型动态配置
- 支持新增自动化场景与文件模板

### 13.3 性能
- 常规请求快速响应，复杂任务先回执后执行
- 支持多子任务并发调度
- 控制共享文件读写开销与任务编排延迟

### 13.4 安全性
- 外部高风险动作需要人工确认
- 子智能体权限应最小化
- 敏感信息写入和读取需可控
- 文件共享应避免误覆盖与越权访问

### 13.5 可观测性
- 任务状态可追踪
- 关键调度节点可记录
- 自动化任务需有执行日志与失败原因

### 13.6 可维护性
- 目录结构、角色定义与文件命名需要标准化
- 便于调试任务路由、子任务失败、记忆污染等问题

## 14. 风险与约束

### 14.1 主要风险
1. **任务拆解不准确**：导致错误派发或重复劳动
2. **上下文碎片化**：文件交接不足时，子智能体理解偏差增大
3. **记忆污染**：把短期噪音写入长期记忆，影响后续判断
4. **过度自动化**：系统在缺少确认时执行高风险动作
5. **通知疲劳**：Heartbeat 过多会降低用户信任
6. **模型能力不均衡**：不同角色模型选择不当影响结果质量

### 14.2 应对策略
- 为任务拆解建立模板与分类规则
- 强制任务 brief 标准化
- 建立 MEMORY.md 写入准则
- 对外部动作增加审批闸门
- 为 Heartbeat 设定频率与阈值策略
- 按任务类型维护模型路由策略

## 15. 版本规划建议

### Phase 1：MVP
- Telegram 单入口
- 主智能体意图识别与基础任务路由
- 2 至 3 个核心角色子智能体
- Shared Workspace 文件协作
- 基础 MEMORY.md 机制
- 基础 Cron 与 Heartbeat 能力

### Phase 2：增强版
- 更完善的任务状态管理
- 子任务依赖编排与失败恢复
- 更多专业角色模板
- 更智能的主动汇报策略
- 自动化任务模板库

### Phase 3：规模化演进
- 可视化任务看板
- 角色市场与技能插件化
- 多项目上下文隔离
- 更细粒度的权限与审计能力

## 16. MVP 验收标准
MVP 发布应至少满足以下条件：
1. 用户可通过 Telegram 向主智能体发送任务并收到结果
2. 主智能体可对复合任务进行拆解，并至少派生两个不同职责子智能体
3. 子智能体可通过共享文件系统进行结果交接
4. 主智能体可读取多个结果并生成统一回复
5. 系统支持至少一种定时任务和一种主动 Heartbeat 汇报
6. 关键用户偏好与项目背景可写入并读取 `MEMORY.md`

## 17. 典型用户故事
1. 作为一名独立开发者，我希望只通过一个 Telegram 聊天窗口就能安排不同职能工作，这样我不需要管理多个 AI 工具。
2. 作为一名 Solo Founder，我希望系统能自动把复杂目标拆分给开发、营销和商务角色，这样我能专注于决策而不是事务调度。
3. 作为一名繁忙的创业者，我希望每天自动收到简洁的关键进展汇报，这样我能及时掌握项目状态。
4. 作为用户，我希望系统能记住我的长期偏好和项目背景，这样我不必反复解释上下文。
5. 作为用户，我希望高风险外部动作必须经过确认，这样我可以放心让系统自动化执行内部工作。

## 18. 结语
Solo Founder Virtual Team 的本质，是将 OpenClaw 从“单一助手”升级为“可编排的虚拟组织系统”。它不是简单地增加多个 Agent，而是通过统一入口、主智能体调度、隔离执行、文件化协作、记忆沉淀与自动化引擎，构建一个可持续运转的个人创业操作系统。

对于 Solo Founder 而言，这将显著降低认知切换成本，提升多线程执行能力，并让个人具备接近小团队的协同产出效率。
